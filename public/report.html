1. 프로젝트 개요

1.1 프로젝트 소개

본 프로젝트는 유리 제조·납품 기업인 동일유리의 실제 업무 프로세스를 면밀히 분석하고, 이를 바탕으로 현장에서 즉시 활용 가능한 **웹 기반 생산관리 시스템(ERP)**을 설계·구현한 것이다.

동일유리가 실제 업무에서 사용하던 엑셀 발주서 현황(2) 파일을 출발점으로 삼아, 해당 데이터를 분석·해석하여 발주 접수부터 생산, 출고, 현장 납품에 이르는 전 과정을 디지털 환경으로 전환하였다. 본 시스템의 지향점은 단순한 데이터 입력 도구가 아니라, 현장의 실시간 상황을 충실히 반영하여 생산팀과 시공팀이 동일한 정보를 공유하며 협업할 수 있는 현장 중심형 통합 플랫폼을 구현하는 데 있었다.

2. 개발 배경 및 문제 정의

2.1 현행 업무 방식의 문제점

동일유리는 유리 제품의 발주 접수부터 현장 납품에 이르는 전 과정을 소수의 관리자가 엑셀 파일에 직접 입력하여 관리해 왔다. 이러한 방식은 관리자 개인의 작업량에 의존하기 때문에 효율성과 편의성이 떨어지고, 입력·수정 과정에서 오류가 발생할 가능성도 높았다. 또한 데이터가 특정 담당자의 파일에 묶여 있어 정보의 단절과 추적의 어려움을 초래하였으며, 그 결과 다음과 같은 구조적 문제가 반복적으로 발생하였다.

문제 영역 현행 문제
발주 현황 파악 담당자 부재 시 진행 상황 확인 불가
생산 일정 관리 생산조별 일정 충돌 및 누락 빈발
납품 지연 대응 지연 발생 시 원인 추적 불가
현장 이동 현황 출고 후 배송·도착 여부를 전화로만 확인 가능
불량·반품 처리 수기 기록으로 이력 관리 불투명
거래처 정보 관리 담당자 개인 연락처에 분산 관리

위 문제들은 개별 사안에 그치지 않고, 정보가 특정 담당자에게 종속되어 조직 차원의 공유와 의사결정을 저해한다는 공통된 원인을 갖는다.

2.2 프로젝트의 방향

본 프로젝트는 엑셀 발주서 현황 파일을 단순히 전산화하는 수준에 머무르지 않고, 생산팀과 시공팀이 현장 상황을 실시간으로 공유할 수 있는 통합 플랫폼을 구현하는 것을 핵심 방향으로 설정하였다. 즉, 정보의 디지털화를 넘어 정보의 흐름과 가시성을 확보함으로써, 현장의 실제 문제를 근본적으로 해소하는 데 목표를 두었다.

3. 기능 요구사항

3.1 인증 & 권한 관리

  • 회원가입 / 로그인 / 로그아웃
  • 역할 기반 접근 제어 (RBAC, 사용자 역할에 따라 사용 가능한 기능을 제한하는 방식): ADMIN / MANAGER / SITE_MANAGER / USER
  • 계정 승인 프로세스 (가입 후 PENDING → 관리자 ACTIVE 처리)
  • 로그인 실패 횟수 제한 및 계정 잠금

3.2 발주 관리

  • 발주 등록 / 조회 / 수정 / 삭제
  • 의뢰번호 자동 채번 (연도-순번 형식, 예: 26-0001)
  • 유리 종류 선택 (TPS / 접합 / 3복층 / 단판 / 기타)
  • 11단계 상태 추적 (대기 → 수령완료 / 불량 → 반품 공장도착)
  • 상태 변경 시 활동 로그 자동 기록
  • 발주서 A4 인쇄 출력

3.3 생산 조(Lot) 관리

  • 발주별 생산조 배치 (1조, 2조, 3조, A조, B조, C조)
  • 조별 수량, 예정일, 완료일, 비고 입력
  • 완료 처리 시 시각적 구분 (초록색 표시)
  • 완료 취소 기능

3.4 일정 관리

  • 주문서 도착일 / 생산의뢰일 / 납품요청일 관리
  • 생산 캘린더: 월별 생산조 일정 시각화
  • 파란=예정 / 초록=완료 / 빨간=지연 색상 구분
  • 캘린더 날짜 클릭 시 해당 발주 상세 이동

3.5 생산 & 출고 처리

  • 상태 변경에 따른 관련 날짜 자동 기록
  • 이동 현황 단계: 발주 → 생산 → 출고 → 배송 → 도착
  • 불량 발생 시 처리 흐름: 불량/반품 → 반품 공장도착
  • 보류 처리 기능

3.6 대시보드 & 분석

  • KPI 카드 4종: 진행 중인 발주 / 이번 주 납품 / 지연 건수 / 수령완료율
  • KPI 카드 클릭 시 발주 목록 자동 필터링 연동
  • 오늘 납품 예정 발주 알림 (당일 납품요청일 기준)
  • 오늘 생산 예정 조 알림 (당일 scheduledDate 기준)
  • 상태별 현황 분포 바 (11개 상태 색상 시각화)
  • 월별 발주 추이 막대 그래프 (최근 6개월)
  • 거래처 Top 5 발주량 바 차트
  • 유리 종류별 발주 분포
  • 지연 발주 목록 (납품요청일 초과 + 미완료)

3.7 거래처 관리

  • 거래처 목록 조회 / 등록 / 수정 / 삭제
  • 발주 등록 시 거래처 자동완성 검색
  • 거래처 상세 페이지: 발주 이력 전체 조회
  • 거래처별 통계: 총 발주 건수, 총 수량, 완료 건수, 자주 쓰는 유리 종류

3.8 알림 & 커뮤니케이션

  • 납품일 기준 지연 발주 대시보드 경고
  • 활동 이력 로그 (STATUS_CHANGED, ORDER_CREATED 등 22가지 이벤트)

5. 기술 스택

5.1 Vercel 최적화 풀스택 구성

본 시스템은 별도의 서버 인프라 구축 없이 단일 프로젝트 내에서 프론트엔드와 백엔드를 통합 운영할 수 있도록, Vercel 생태계에 최적화된 풀스택 아키텍처로 구성하였다.

Frontend     Next.js 16 (App Router)   서버 사이드 렌더링, 파일 기반 라우팅 (App Router: 페이지 경로 관리 방식)
UI           Tailwind CSS + shadcn/ui   일관된 디자인 시스템, 빠른 UI 구성
Backend      Next.js API Routes         별도 서버 불필요, 풀스택 단일 프로젝트
Database     PostgreSQL (Supabase)      안정적 관계형 DB, 클라우드 관리형
ORM          Prisma 7                   타입 안전 DB 쿼리, 스키마 마이그레이션 (ORM: 데이터베이스 연동 도구 / 스키마: 데이터베이스 구조 설계도)
인증         세션 기반 자체 인증         역할 기반 접근 제어 (RBAC)
배포         Vercel                     자동 CI/CD, 글로벌 CDN

5.2 선택 이유

각 기술은 4주라는 제한된 개발 기간 내에서 안정성과 생산성을 동시에 확보한다는 기준에 따라 선정하였다.

  • Next.js: 서버 컴포넌트를 활용하여 초기 데이터 로딩을 서버 측에서 처리함으로써, 사용자에게 빠른 화면 응답을 제공한다.
  • Prisma: 데이터 모델을 코드로 관리(Schema as Code)하고 타입 안전성을 보장하여, 런타임 오류를 사전에 차단한다.
  • Supabase: 별도의 데이터베이스 서버 구성 없이 관리형 PostgreSQL을 즉시 사용할 수 있어, 개발 속도를 크게 향상시킨다.
  • Vercel: Git push만으로 자동 배포가 이루어지며, 팀 협업 시 각 브랜치별 미리보기 URL이 자동 생성되어 협업 효율을 높인다.

6. 데이터베이스 설계

6.1 Prisma 스키마 모델 구조

데이터베이스는 사용자(User), 거래처(Client), 발주(Order)를 중심으로, 발주에 종속되는 생산조(ProductionLot)와 활동 로그(Activity)가 유기적으로 연결되는 구조로 설계하였다. 이를 통해 발주를 기준으로 한 모든 정보가 단일 흐름으로 추적 가능하도록 하였다.

User (사용자)
├── id, username, passwordHash, fullName
├── department, phone
├── role: ADMIN / MANAGER / SITE_MANAGER / USER
├── status: PENDING / ACTIVE / INACTIVE
└── failedLogins, lockedUntil (보안 처리)

Client (거래처)
├── id, name (unique), shortCode
├── phone, memo
└── orders: Order[] (발주 이력 연결)

Order (발주)
├── orderNumber (자동 채번: 26-0001 형식)
├── clientName, clientId (거래처 연결)
├── siteName, quantity, area
├── frameType, glassType (TPS/LAMINATED/TRIPLE/SINGLE/OTHER)
├── productName, noteDefect, noteJoint
├── 일정 필드 9개 (orderReceivedDate ~ returnReceivedDate)
├── status: OrderStatus (11단계)
├── createdById, assignedUserId
└── productionLots: ProductionLot[]
    activities: Activity[]

ProductionLot (생산조)
├── orderId (발주 연결)
├── lotName (1조, 2조, A조 등)
├── quantity, scheduledDate, completedDate
└── note

Activity (활동 로그)
├── orderId, userId
├── action: ActivityType (22가지 이벤트)
└── changes: Json (변경 전/후 데이터 저장)

6.2 발주 상태 흐름도 (OrderStatus)

발주의 생애주기는 생산 단계와 물류 단계, 그리고 불량·보류 등의 예외 처리를 포괄하는 11단계 상태로 정의하였다. 이는 현장의 실제 업무 흐름을 그대로 모델에 반영한 결과이다.

[대기] ──▶ [생산 준비] ──▶ [생산 중] ──▶ [생산완료] ──▶ [출고]
                                                            │
                                          [보류] ◀──────────┤
                                                            │
                                                     [배송 중] ──▶ [도착] ──▶ [수령완료]
                                                            │
                                                     [불량/반품] ──▶ [반품 공장도착]

정상 흐름: 대기 → 수령완료 (7단계)
불량 흐름: 출고 이후 불량 발생 → 반품 → 공장 도착 (2단계 추가)

7. 프론트엔드 구조

7.1 페이지 구성

화면은 업무 단위에 따라 명확히 분리하여, 사용자가 자신의 역할과 작업 맥락에 맞는 페이지에 직관적으로 접근할 수 있도록 구성하였다.

경로 페이지 주요 기능
/dashboard 대시보드 KPI 카드, 상태 분포, 지연 목록

/orders/new 발주 등록 거래처 자동완성, 유리종류 선택
/orders/[id] 발주 상세 생산조 관리, 활동 이력
/orders/[id]/edit 발주 수정 상태 직접 변경 (ADMIN/MANAGER)
/orders/[id]/print 발주서 인쇄 A4 최적화 출력
/calendar 생산 캘린더 월별 생산조 일정
/clients 거래처 목록 등록, 조회, 관리
/clients/[id] 거래처 상세 발주 이력, 통계
/admin/users 사용자 관리 계정 승인, 역할 변경

7.2 핵심 컴포넌트

재사용성과 유지보수성을 고려하여 주요 기능을 독립적인 컴포넌트로 모듈화하였다.

OrderDataTable (발주 목록)

  • 상태 드롭다운 필터 (전체·진행중·11개 상태 선택)
  • 업체명 / 날짜 범위 / 키워드 검색
  • 컬럼: 의뢰번호, 업체명, 현장명, 유리종류, 수량, 납품요청일, 상태
  • 지연 발주 배지(badge) 자동 표시

OrderForm (발주 등록/수정)

  • 업체명 자동완성 Combobox (거래처 DB 연동)

8.2 단계별 전용 라우트 설계 방식

각 상태 전이를 하나의 PATCH로 처리하지 않고 단계별 전용 라우트로 분리한 이유는, 상태와 날짜 기록의 책임을 명확히 분리하기 위함이다. 예를 들어 /preparing 라우트는 오직 WAITING → PREPARING 전이와 preparingDate 기록만을 담당하며, 잘못된 상태에서의 호출은 서버에서 즉시 차단한다.

상태 되돌리기(revert) 로직: 상태를 이전 단계로 복원할 때는 해당 날짜 필드를 null로 초기화하여 데이터 정합성을 보장한다.

const revertMap = {
  PREPARING:       { prev: "WAITING",        clearField: "preparingDate" },
  PRODUCTION:      { prev: "PREPARING",      clearField: "productionDate" },
  PRODUCTION_DONE: { prev: "PRODUCTION",     clearField: "productionDoneDate" },
  SHIPPED:         { prev: "PRODUCTION_DONE", clearField: "shipmentDate" },
  IN_DELIVERY:     { prev: "SHIPPED",        clearField: "deliveryStartDate" },
}

모든 상태 전이는 Activity 테이블에 자동 기록되어 변경 이력 추적이 가능하다.

9.2 중간 발표 및 피드백

기반 시스템을 완성한 시점에서 진행된 중간 발표는, 본 프로젝트의 방향성을 재정립하는 중요한 전환점이 되었다. 교수님과 심사자들은 시스템의 완성도를 인정하면서도, 다음과 같은 핵심적인 문제를 지적하였다.

"발주서 현장 데이터를 활용하지 못하여 현실감이 떨어진다. 현장 상황을 보다 빠르고 정확하게 파악하는 것이 핵심이다."

이 지적은 곧, 우리가 구현한 시스템이 기능적으로는 충실하지만 정작 '현장의 살아 있는 흐름'을 담아내지 못하고 있다는 본질적인 한계를 짚은 것이었다. 심사자들은 구체적으로 다음 세 가지 개선 방향을 요구하였다.

  1. 실제 데이터: 가상의 임의 데이터가 아닌, 현장감이 살아 있는 실제 발주 흐름 데이터
  2. 현재 이동 현황: 현장에서 전화로 문의하지 않고도 발주의 위치를 파악할 수 있는 상태 추적
  3. 앱 기능: 현장에서 직접 사진을 촬영하여 불량을 등록할 수 있는 기능

9.3 개발 2단계: 피드백 반영 및 개선

팀은 이 피드백을 단순한 보완 요청이 아니라 프로젝트의 정체성을 재정의하는 과제로 받아들였다. "기능이 많은 시스템"에서 "현장의 실제 문제를 푸는 시스템"으로 무게중심을 옮기는 것을 2단계의 목표로 삼고, 시스템의 현장성을 전면적으로 강화하였다.

① 예시 데이터 구성

기업에서 제공받은 엑셀 파일을 분석한 결과, 날짜가 이미 경과되었거나 항목이 비어 있는 데이터가 많아 이를 그대로 시스템에 적용하기 어려웠다. 이에 따라 현장의 실제 업무 흐름을 반영한 예시 데이터를 별도로 구성하였다. 발주 등록부터 납품 완료까지 각 단계별 시나리오를 담은 20건의 데이터를 직접 생성함으로써, 시스템이 현장에서 어떻게 활용될 수 있는지를 구체적으로 시연할 수 있도록 하였다.

② 이동 현황 11단계 상태 고도화

기존의 단순한 상태 관리 체계를 실제 물류 흐름을 반영한 11단계 상태 체계로 전면 개편하였다.

생산팀 관점: 대기 → 생산 준비 → 생산 중 → 생산완료
시공팀 관점: 출고 → 배송 중 → 도착 → 수령완료
예외 처리: 불량/반품 → 반품 공장도착
보류 처리: 언제든 보류 전환 가능

이러한 관점 분리를 통해 생산팀은 생산 진행 상황을, 시공팀(현장)은 납품 이동 현황을 각자의 업무 시점에서 실시간으로 확인할 수 있게 되었다. 이는 "현장에 전화하지 않아도 발주의 위치를 알 수 있어야 한다"는 피드백에 대한 직접적인 해답이었다.

③ 데이터 통합 연결 강화

발표 준비 과정에서, 대시보드·발주 목록·생산 캘린더·거래처 관리 사이의 데이터가 일관된 기준으로 연결되어 있지 않다는 문제를 발견하였다. 예컨대 대시보드에 표시된 "진행 중인 발주 18건"을 클릭하여 발주 목록으로 이동하면 화면마다 집계 기준이 달라 서로 다른 수치가 나타나는 결함이 있었다. 이를 해소하기 위해 모든 화면에 동일한 집계 기준을 일괄 적용하고, 대시보드 KPI 클릭 시 해당 조건으로 필터링된 발주 목록으로 자동 이동하는 연동을 완성하였다.

④ 권한 기반 상태 관리

마지막으로, 상태 변경 권한을 ADMIN/MANAGER에게만 부여하는 역할 기반 접근 제어를 강화하였다. 일반 USER는 조회 및 기본 등록만 수행할 수 있도록 권한을 명확히 구분함으로써, 무분별한 상태 변경으로 인한 데이터 신뢰성 훼손을 방지하였다.

이상의 개선을 통해, 본 시스템은 단순한 디지털 발주서를 넘어 현장의 흐름을 실시간으로 반영하는 플랫폼으로 거듭났다.

9.4 개인 소견

프로젝트를 진행하면서 개인적으로 가장 많이 생각한 부분은 기업과 실제 사용자의 입장이었다. 기존에 쓰던 엑셀 방식을 새로운 웹 서비스로 바꾸려면, 그만한 이유가 있어야 한다고 생각했다. 기업 전체가 함께 쓰는 서비스인 만큼 누구나 쉽고 편하게 사용할 수 있어야 하고, 화려한 기능보다는 정말 필요한 데이터를 한눈에 볼 수 있는 것이 더 중요하다고 판단했다.

최종 발표를 준비하면서 이러한 생각이 잘 드러날 수 있도록 예시 데이터를 구성하며 시스템을 다듬었고, 그 과정에서 모바일로도 쓸 수 있다면 현장에서의 활용도가 훨씬 높아질 것이라는 생각이 들었다. 이동 현황 추적이나 음성 AI, 카카오 알림 같은 기능이 더해진다면 현장성과 데이터 관리 측면 모두에서 더 완성도 있는 서비스가 될 수 있을 것이라 생각한다.

10. 주요 기술 구현 상세

10.1 의뢰번호 자동 채번

발주마다 고유한 식별자를 부여하기 위해, 연도와 순번을 결합한 형식의 의뢰번호를 자동 생성하도록 구현하였다. 직전 발주 번호를 조회하여 순번을 1 증가시키는 방식으로, 중복 없이 순차적으로 채번된다.

async function generateOrderNumber(): Promise<string> {
  const yy = String(new Date().getFullYear()).slice(-2)  // "26"
  const last = await prisma.order.findFirst({
    where: { orderNumber: { startsWith: `${yy}-` } },
    orderBy: { orderNumber: "desc" },
  })
  if (!last) return `${yy}-0001`
  const seq = parseInt(last.orderNumber.split("-")[1] ?? "0", 10)
  return `${yy}-${String(seq + 1).padStart(4, "0")}`  // "26-0001"
}

10.2 역할 기반 상태 수정 제어

데이터의 무결성을 보장하기 위해, 이미 완료 처리된 발주는 ADMIN/MANAGER 권한이 없는 사용자가 수정할 수 없도록 서버 측에서 차단한다.

const ADMIN_ROLES = ["ADMIN", "MANAGER"]
if (!ADMIN_ROLES.includes(session.role) 
    && ["RECEIVED", "RETURN_RECEIVED"].includes(order.status)) {
  return Response.json({ error: "완료된 발주는 수정할 수 없습니다." }, { status: 400 })
}

10.3 대시보드 필터 통일

화면 간 집계 불일치 문제를 근본적으로 해소하기 위해, '완료' 상태의 정의를 단일 상수로 통일하고 진행 중·지연 발주의 산정 로직을 일관되게 적용하였다.

const DONE_STATUSES = ["RECEIVED", "RETURN_RECEIVED", "HOLD"]
const inProgress = orders.filter(
  (o) => !([...DONE_STATUSES, "SHIPPED"]).includes(o.status)
).length
const delayed = orders.filter(
  (o) => o.deliveryRequestDate
    && isBefore(new Date(o.deliveryRequestDate), today)
    && !DONE_STATUSES.includes(o.status)
).length

요컨대 본 프로젝트는 현장의 구체적 문제 정의에서 출발하여, 중간 발표의 피드백을 정면으로 수용·반영하고, 이를 운영 가능한 실물 시스템으로 완성하였다는 점에서 실무적 의의를 갖는다.

12. 향후 발전 방향 (Road Map)

본 시스템은 현장 밀착형 서비스로서의 가능성을 입증하였으며, 여기에 다양한 기능을 더해 나간다면 그 가치를 한층 확장할 수 있다.

우선 모바일 접근성 측면에서, 시스템을 PWA(스마트폰 홈 화면에 설치해 앱처럼 사용하는 웹 기술) 형태로 제공하면 현장 작업자가 별도의 앱 설치 없이 스마트폰으로 손쉽게 이용할 수 있다. 기업 전체가 함께 쓰는 서비스일수록 진입 장벽이 낮아야 한다는 점에서, 모바일 최적화는 가장 우선적으로 고려할 방향이다.

현장성 강화 측면에서는 배송 차량의 GPS 위치 추적을 연동하여 유리가 지금 어디에 있는지를 지도 위에서 실시간으로 확인할 수 있고, 카카오 알림톡을 통해 상태가 변경될 때마다 현장 담당자에게 즉시 메시지를 전달하는 기능도 구현 가능하다. 더불어 음성 AI를 활용하면 현장에서 손을 쓰기 어려운 상황에서도 음성으로 발주 상태를 조회하거나 메모를 남기는 등 입력 편의성을 크게 높일 수 있다. 불량 발생 시 사진을 직접 촬영하여 등록하는 기능이 더해진다면, 불량 처리 이력을 보다 신속하고 투명하게 관리할 수 있을 것이다.

데이터 고도화 측면에서는 기업의 기존 ERP(전사적 자원관리 시스템)와 재고·회계 데이터를 연동하거나, 과거 발주 데이터를 학습한 AI 수요 예측을 통해 생산 계획 수립을 지원하는 방향도 고려할 수 있다.

이러한 기능들이 추가된다면 시간과 노력이 더 필요하겠지만, 현장성과 데이터 관리 측면 모두에서 한층 완성도 있는 서비스로 발전할 수 있을 것이라 판단된다.
